# CLI Architecture

Concrete, stdlib-only layout used by `mm-crypt-cli`. No third-party CLI framework —
every command group hangs off `argparse` directly, so the installed dependency
graph stays minimal (important for a security tool).

## Project layout

```
src/mm_crypt_cli/
├── __init__.py
├── errors.py                   # CliError (recoverable error, (message, code))
├── main.py                     # argparse parser + `app(argv)` dispatcher
├── io.py                       # stdin/stdout/file read+write helpers
├── secrets.py                  # resolve_secret() — --key | --key-file | --key-env
├── commands/
│   ├── editor.py               # `mm-crypt editor <path>` — top-level TUI command
│   ├── fernet/                 # `mm-crypt fernet  (f) ...`
│   │   ├── keygen.py           # keygen (g)
│   │   ├── encrypt.py          # encrypt (e)
│   │   └── decrypt.py          # decrypt (d)
│   ├── openssl/                # `mm-crypt openssl (o) ...`
│   │   ├── encrypt.py          # encrypt (e)
│   │   └── decrypt.py          # decrypt (d)
│   └── scrypt/                 # `mm-crypt scrypt  (s) ...`
│       ├── encrypt.py          # encrypt (e)
│       └── decrypt.py          # decrypt (d)
└── simpletui/                  # Hand-rolled termios TUI library (no third-party deps)
```

Flat layout: `main.py`, `io.py`, `secrets.py`, `errors.py` sit at the package
root next to `commands/` and `simpletui/`. No `cli/` wrapper layer, no
`core/` composition root — the crypto commands hold no persistent state, so
the ceremony earns nothing. `simpletui/` is kept as its own package because
it's a self-contained editor library that may eventually graduate to a
standalone PyPI package.

## Command module shape

Each command file exposes exactly two symbols:

```python
import argparse

def register(subparsers: argparse._SubParsersAction[argparse.ArgumentParser]) -> None:
    """Add this command's subparser under the given group."""
    parser = subparsers.add_parser("encrypt", aliases=["e"], help="...", description="...")
    parser.add_argument("--key", "-k", default=None, help="...")
    # ...
    parser.set_defaults(func=_run)

def _run(args: argparse.Namespace) -> None:
    """Do the actual work. Raise CliError on recoverable failures."""
    ...
```

`main.py` builds the top-level parser, creates one sub-app per group
(`fernet`, `openssl`, `scrypt`), calls `<module>.register(group_subparsers)`
for every nested command, and registers top-level commands like `editor`
directly on the outer subparsers. Dispatch is `args.func(args)`.

## Error handling

`CliError(message, code)` (in `mm_crypt_cli.errors`) is raised from anywhere
inside a command's `_run`. `main.py:app()` catches it at the top, writes
`Error: <message> [<CODE>]` to stderr, and returns exit code 1. Every other
`SystemExit` (argparse usage errors → 2, `--help` / `--version` → 0) flows
through unchanged.

Use UPPER_SNAKE_CASE codes: `MISSING_SECRET`, `DECRYPTION_FAILED`, `INVALID_INPUT`, etc.

## Entry point

`pyproject.toml`:

```toml
[project.scripts]
mm-crypt = "mm_crypt_cli.main:app"
```

`app(argv: list[str] | None = None) -> int` returns the process exit code; the
console-script wrapper generated by the build backend calls `sys.exit(app())`.

## Testing

`mm-crypt-cli/tests/conftest.py` provides a tiny in-house `CliRunner` + `Result`
pair that mimics the `typer.testing.CliRunner` API we used to depend on. Tests
call `runner.invoke(app, ["fernet", "encrypt", "--key", key], input="hello")`
and assert on `result.exit_code`, `result.stdout`, `result.stdout_bytes`, and
`result.stderr` (or the combined `err_text` fixture).

## Rules

1. No third-party CLI framework. Stdlib `argparse` only.
2. Per-command files in `commands/<group>/<verb>.py` for grouped commands, or
   `commands/<verb>.py` for top-level commands. One file per command.
3. Each command exposes `register(subparsers)` + `_run(args)` — main.py stays thin.
4. Raise `CliError(message, code)`; never `sys.exit()` directly from a command.
5. `__init__.py` files stay empty (module docstring only) — no re-exports, no logic.
6. `pydantic` is not a dependency of `mm-crypt-cli`. Data classes use stdlib
   `@dataclass(frozen=True, slots=True)`.
